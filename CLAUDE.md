# CLAUDE.md

> 张文冠的个人网站。本文件是项目的约束与原则来源。Claude Code 在本项目内的所有改动都应遵守本文件。
>
> 设计部分刻意只给**目标与原则**，不锁死配色 / 版式 / 组件结构——视觉方案应在会话中探索、提案、确认后再落地，避免被既有思路框住。

---

## 1. 这是什么网站（最高准则）

这是**张文冠的个人空间**，不是求职作品集。

一句话目标：**让来逛的人感觉到「这是一个具体的、有点意思的人」，待着舒服、有记忆点。**

- 访客离开时记住的应该是**这个人的气质与趣味**，而不是一份技能清单或 KPI。
- 技术项目仍然展示，但作为「**我做过 / 我感兴趣的东西**」来叙述，而不是用来打动招聘方的证据。
- 基调：第一人称、松弛、有温度、带点个人趣味，但「松弛」不等于敷衍或卖萌——**真诚、具体、有自己的声音**才是目标。

**受众**
- 主要：来逛逛的人、朋友、同好
- 次要：海外访客（通过英文切换访问）

**内容优先级**
1. 自我介绍 / 自述——「我是谁、我在乎什么」，带个人味道
2. 兴趣 / 生活 / 想法——爱好、读过的东西、随笔等更私人的内容（板块可先占位，内容后续再填）
3. 做过的东西（智能客服小丰、dodo-Agent 等，弱化求职感的叙述）
4. 教育与经历作为背景信息，不堆砌为卖点

## 2. 个人信息与项目数据（事实来源）

所有事实数据（姓名、职衔、联系方式、教育、荣誉、项目）统一放在 **`src/data/profile.json`**，需要时读取，不在本文件或组件里重复硬编码。

- 数据结构为「字段 → `{ zh, en }`」的双语形式；不翻译的字段（如 `dodo-Agent`）两版保持一致。
- 这些是**事实，不可篡改**。新增 / 修改内容只改这一份 JSON，不在各处散落副本。
- ⚠️ **项目 highlight 的技术细节必须使用简历原句，不得二次总结 / 精简**。可在 highlight 外围加个人化的引子或感受，但 highlight 本体保持原句；英文版为忠于原句的完整翻译（术语大小写按官方写法）。
- 在 Next 里可直接 `import profile from "@/data/profile.json"`（`tsconfig` 已开 `resolveJsonModule`），或在数据层封装后供组件取用。

## 3. 技术栈（核心固定，其余按需引入）

**核心（固定，不替换）**

| 类别 | 选型 | 说明 |
|---|---|---|
| 框架 | Next.js (App Router) | SSR/SEO 友好，Vercel 部署零配置 |
| 语言 | TypeScript (strict) | 禁用 `any`，用 `unknown` 或具体类型 |
| 样式 | Tailwind CSS v4 | 无 `tailwind.config.ts`，token 走 `globals.css` 的 `@theme` / `:root` |
| 部署 | Vercel | |

**其余一律「按需引入，能不加就不加」**——每加一个依赖，先问一句：「这个记忆点 / 体验，值不值这点体积与复杂度？」保持 bundle 精简、首屏轻快。

- **动画**：先用 CSS / Tailwind transition 解决；需要入场编排、滚动驱动、共享布局动画（`layoutId`）时，再引入 **Framer Motion**。
- **交互组件**：需要复杂交互（弹层、下拉、表单、对话框等）时再 `npx shadcn add` 对应组件，按需取用、可深度定制。
- **现成视觉效果**：想要某个特定效果时，从 **Magic UI**（`magic-ui` MCP）/ **react-bits**（`react-bits` MCP）单独取**那一个**组件复制进项目即可，不必整库依赖；注意适配主题色与浅/暗底，依赖 `motion/react` 的改用 `framer-motion`。
- **图标**：需要时用 lucide-react。
- **3D**：仅当某个记忆点确实需要，才用 `@react-three/fiber` + `@react-three/drei`；`next/dynamic({ ssr:false })` 局部挂载 + 懒加载，绝不拖累首屏。
- **视频化动画（Remotion）**：体积大、加载慢，**默认不用**；能用 Framer Motion 或一段视频替代的就不上。

**判断准则**：核心三件套之外的任何依赖，都应由「当前这个具体需求」触发，而不是预先铺好。先确认需要，再引入。

**禁止引入**：Redux、MobX、styled-components、emotion、Material UI、Ant Design。本项目规模不需要。

## 4. 设计：目标与原则（不锁死方案）

设计方向**开放探索**。动手前先就「调性 / 配色 / 版式 / 关键动效」给 1-2 句提案，确认后再实现。下列是原则与底线，不是模板：

**应该追求**
- **个人气质优先**：第一眼能感到「一个具体的人」，而非通用 SaaS / 求职模板。版式、配色、字体、文案都为这个目标服务。
- **舒服耐看**：留白充足、阅读节奏好、层级清晰；色调与气质统一、不杂乱。
- **有记忆点但不炫技**：可以有一两处让人会心一笑或眼前一亮的细节（动效、版式、插画、3D 局部等），但服务氛围、不喧宾夺主、不拖慢首屏。
- **松弛中的考究**：字号阶梯拉得开、行高字距讲究、中英文混排有节奏。

**底线（硬约束）**
- **可读性 / 对比度**：正文与背景对比必须达标，任何配色方案都不能牺牲正文可读性。正文字号不小于 16px（`text-base`），禁止小于 12px。
- **性能**：Lighthouse Performance > 90、Accessibility > 95 为目标；首屏不加载重资源（3D / 大动画动态 import）；图片用 `next/image` 并带宽高；字体用 `next/font`。
- **响应式**：移动优先，375 / 768 / 1280 三种宽度都要成立。
- **节制**：一屏内配色、字重、动效种类都要克制；宁可少而准。
- **token 化**：颜色集中在 `globals.css` 的 `@theme` / `:root`，组件内不写未定义的颜色字面量，方便整体换肤。

**动画原则**
- 滚动进入视口：`opacity 0→1 + 轻微 y 位移`，时长 ~0.5s，缓动偏 ease-out。
- 悬停：轻微 `scale` / 抬升 + 阴影，过渡 ~0.2s。
- 语言切换：整页淡出淡入 ~0.2s。
- 文字揭起类 `overflow-hidden` 包裹动画，必须为下伸字形（`g`/`y`/`p`）留 `pb`（如 `pb-[0.18em]`，外层等量 `-mb` 抵消），否则英文字母下缘被裁。
- 默认避免：粒子背景、全屏 3D 旋转、夸张视差、过度 spring 抖动——除非明确服务于某个记忆点且不伤性能。

## 5. 国际化与翻译 ⚠️ 重点

### 5.1 默认语言与持久化
默认中文。首次访问检测 `navigator.language`，非 `zh-*` 默认英文。用户手动切换后写入 `localStorage`，下次访问尊重用户选择。切换时整页淡出淡入。

### 5.2 翻译原则（通用）
- **技术专有名词在中英两版都保持官方原文与官方大小写**，不音译、不意译。覆盖：编程语言 / 运行时、框架 / 库、数据库 / 中间件、算法 / 模型、大模型概念与缩写、并发概念、模型产品名、流式 API、资质证书等。
  - 示例（以官方写法为准，拿不准时查官方）：`Spring Boot`、`Elasticsearch`、`ONNX Runtime`、`BGE-RERANKER`、`RAG`、`ReAct`、`CoT`、`Few-shot`、`MCP`、`SSE`、`BM25`、`RRF`、`通义千问`、`CET-4`、`ISN`……
  - ✅ 正确：`实现 4 维 Query Rewriting（简洁化/抽象化/纠错/标准化）；查询路由器将 Query 分发至 Elasticsearch / MySQL / Neo4j，失败自动降级。`
  - ❌ 错误：`实现 4 维查询重写……将查询分发至弹性搜索 / 关系数据库 / 图数据库……`
- 大小写遵循官方写法（`Elasticsearch` 不是 `ElasticSearch`，`Spring Boot` 不是 `SpringBoot`）。
- 中文里数字用阿拉伯数字（`4 维`、`98%`），中英文之间留空格（`基于 LLM 实现`）。

### 5.3 人名 / 校名 / 机构
中文版用中文，英文版用**官方英文译名**；缩写（如 `ISN`）两版都保留。具体译名以 `src/data/profile.json` 为准（如：西安电子科技大学 → Xidian University；ISN 国家重点实验室 → State Key Laboratory of ISN）。

## 6. 内容写作风格

> 总基调：个人空间，文字允许第一人称、松弛、有温度、带点趣味，不必端着。但真诚、具体、有声音是底线。

### 6.1 中文
- 自述 / 兴趣 / 想法：第一人称、口语化的真诚表达，像在跟人聊天，但别堆网络梗和无意义语气词。
- 数字用阿拉伯数字；中英文之间加空格。
- 项目叙述为「我做过 / 我感兴趣的东西」，去求职话术，但要具体不空泛；**highlight 本体用简历原句**（见第 2 节）。

### 6.2 英文
- active voice；个人内容可第一人称、轻松；项目技术描述简洁，每句不超过 25 词。
- 技术名词大小写遵循官方写法。

### 6.3 避免
- ❌ "本人"、"该项目"等公文腔 → 个人内容直接用「我」，项目用项目名指代。
- ❌ 求职话术、自夸形容词堆砌（"极强能力"、"完美解决"）→ 用具体的事和真实感受说话。
- ❌ 过度网络梗、卖萌语气词堆砌。
- ✅ Emoji 可在个性内容里克制点缀，不滥用、不每句都加。

## 7. 代码规范

- 组件一律函数组件 + Hooks；默认 `'use client'`，纯展示无交互的才用 Server Component。
- 组件文件名 PascalCase，一个文件一个主组件；props 显式声明 `interface ComponentNameProps`。
- 优先 Tailwind 工具类，避免写 CSS 文件；复用样式用 `cn()` 拼接，不用 `@apply`。
- 响应式移动优先，断点 `md:`(768) / `lg:`(1024)。
- 复杂逻辑写中文注释；不写自解释的废话注释；TS 类型不写 JSDoc。
- 数据从 `src/data/profile.json` / 字典层取，Section 组件只负责布局与动画。

## 8. 沟通规则

1. **先解释方案，再动手**：复杂或涉及视觉方向的改动，先用 1-2 句说明思路，确认后再改文件。
2. **保持上下文**：每次只动相关文件，不顺手重构无关代码。
3. **拒绝过度设计**：个人网站，不需要状态管理库、单元测试、CI/CD 配置。
4. **中文回复**，代码注释也用中文。
5. **保留用户决策**：核心技术栈（第 3 节）不要主动建议替换；新依赖按「确认需要再引入」的准则走。
